1. 개요 - 왜 두 가지 방식인가?

DPDK UDP
커널 바이패스 + 유저 스페이스 패킷 처리
극한의 저지연 & 고처리량
VS
Socket UDP
OS 커널 네트워크 스택 활용
범용 호환성 & 간편한 설정

Dpdk_Test_Tool은 동일한 UDP 통신 테스트를 두 가지 완전히 다른 방식으로 수행할 수 있는 비교 도구입니다.

DPDK UDP

NIC 하드웨어를 직접 제어하여 커널을 완전히 우회합니다. 패킷 헤더(Ethernet, IPv4, UDP)를 직접 구성하고, Hugepage 메모리에서 DMA를 통해 NIC와 직접 데이터를 교환합니다.

10M+ PPS, 마이크로초 레이턴시

Socket UDP

Windows 커널 네트워크 스택(TCPIP.SYS)을 통해 패킷을 처리합니다. OS가 ARP, 라우팅, 헤더 구성을 모두 자동으로 처리합니다.

범용 호환, 간편한 설정

두 방식을 동일한 테스트 도구에서 비교 실행하면, DPDK 도입으로 얻는 실제 성능 이득을 정량적으로 측정할 수 있습니다.

2. 아키텍처 비교

DPDK UDP 아키텍처

C# Application DpdkUdpTab / DpdkUdpTxEngine / DpdkUdpRxEngine
↓
P/Invoke (Marshal) Ethernet + IPv4 + UDP 헤더 직접 구성
↓
dpdk_shim.dll (C) mbuf 할당, tx/rx burst, SEH 보호
↓
DPDK Library rte_eth_tx_burst / rte_eth_rx_burst
↓
ixgbe PMD Intel X550 Poll Mode Driver (유저 스페이스)
↓
netuio Driver PCI BAR 매핑 (커널 바이패스)
↓
NIC Hardware Intel X550-AT2 10GbE

Socket UDP 아키텍처

C# Application SocketUdpTab / SocketUdpTxEngine / SocketUdpRxEngine
↓
.NET Socket API Socket(Dgram, Udp) / SendTo / ReceiveFrom
↓
Windows Kernel Winsock → AFD.SYS → TCPIP.SYS
↓
NDIS / miniport 커널 모드 NIC 드라이버
↓
NIC Hardware Intel X550-AT2 10GbE
핵심 차이: DPDK는 커널을 완전히 우회하여 NIC 하드웨어에 직접 접근합니다. 이로 인해 컨텍스트 스위치, 시스템 콜, 인터럽트 처리 오버헤드가 제거됩니다. 반면, Socket은 OS가 제공하는 안전한 추상화 계층을 활용합니다.

3. 패킷 처리 파이프라인

TX (송신) 흐름 비교

DPDK TX Pipeline

// 1. mbuf 메모리 풀에서 버퍼 할당 IntPtr mbuf = shim_pktmbuf_alloc(pool); // 2. 공간 확보 (헤더 + 페이로드) shim_pktmbuf_append(mbuf, totalLen); // 3. Ethernet 헤더 직접 작성 Marshal.Copy(dstMac, 0, ptr, 6); Marshal.Copy(srcMac, 0, ptr+6, 6); Marshal.WriteInt16(ptr+12, 0x0008); // IPv4 // 4. IPv4 헤더 직접 작성 // 5. UDP 헤더 직접 작성 // 6. 페이로드 복사 // 7. 32개 패킷 묶어서 버스트 전송 shim_tx_burst(portId, queueId, mbufs, count);

Socket TX Pipeline

// 1. 소켓 생성 var socket = new Socket( AddressFamily.InterNetwork, SocketType.Dgram, ProtocolType.Udp); // 2. 페이로드만 준비 byte[] payload = Encoding.ASCII .GetBytes(text); // 3. 전송 (OS가 헤더 자동 구성) socket.SendTo(payload, endpoint); // OS가 자동으로 처리하는 것들: // - ARP 해석 // - Ethernet/IP/UDP 헤더 // - 체크섬 계산 // - NIC 드라이버 호출

RX (수신) 흐름 비교

DPDK RX Pipeline

// 1. 폴링으로 패킷 수신 (최대 32개) int count = shim_rx_burst( portId, queueId, mbufs, 32); // 2. 각 패킷 헤더 직접 파싱 for (int i = 0; i < count; i++) { // Ethernet 타입 확인 // IPv4 src/dst 추출 // UDP port 필터링 // 페이로드 복사 } // Busy-wait: CPU 100% 점유하며 폴링

Socket RX Pipeline

// 1. 타임아웃 기반 수신 대기 socket.ReceiveTimeout = 100; // ms // 2. 수신 (OS가 헤더 제거 후 전달) int len = socket.ReceiveFrom( buffer, ref remoteEP); // 3. 페이로드만 수신됨 string data = Encoding.ASCII .GetString(buffer, 0, len); // 이벤트 기반: CPU 효율적

CPU 사용 단일 코어 100% 점유 (busy-wait) 상대적 효율적 (이벤트 기반)
지터(Jitter) 극히 낮음 — 예측 가능한 처리 높음 — 커널 스케줄러, 인터럽트 변동
CPU 트레이드오프: DPDK는 극한의 성능을 위해 CPU 코어 하나를 100% 점유합니다 (Busy-wait 폴링). 이는 처리할 패킷이 없을 때도 CPU를 사용하므로, 에너지 효율은 Socket 방식이 우수합니다. 테스트 도구에서는 PowerPlanHelper를 통해 자동으로 고성능 전원 설정을 적용하여 C-State 진입을 방지합니다.

6. 동작 모드

공통 모드 (DPDK + Socket)

TX (송신)

지정된 대상으로 UDP 패킷을 연속 전송합니다. PPS 제한 설정 가능.

DPDK Socket

RX (수신)

지정된 포트에서 UDP 패킷을 수신하고 로그를 기록합니다.

DPDK Socket

Echo

수신한 패킷을 그대로 송신자에게 반송합니다. 레이턴시 측정용.

DPDK Socket

Req/Resp

요청 전송 후 응답 대기. RTT (Min/Max/Avg/StdDev) 측정.

DPDK Socket

DPDK 전용 기능

Pipeline 모드

Window Size > 1 설정 시, 응답을 기다리지 않고 N개의 요청을 연속 전송. 배치 통계(shim_reqresp_batch)로 처리량 극대화.

DPDK 전용

Warmup

본 측정 전 64개 더미 패킷 전송. NIC DMA, CPU 캐시, TLB를 예열하여 초기 지터 감소.

DPDK 전용

ARP 자동 해석

목적지 MAC이 미지정(FF:FF:FF:FF:FF:FF) 시, ARP Request/Reply로 자동 해석. Gratuitous ARP 3회 전송.

DPDK 전용

Socket 전용 기능

Server 모드

수신 대기 + 자동 응답 텍스트 설정. 수동 메시지 전송 큐도 지원. 마지막 송신자에게 자동 라우팅.

데이터 흐름 요약

DPDK 호출 체인

Socket 호출 체인

SocketUdpTab (UI, WinForms) → SocketUdpTxEngine (C#, Thread) → Socket.SendTo (.NET) → WSASendTo (Winsock2) → AFD.SYS (커널) → TCPIP.SYS (IP/UDP 스택) → NDIS miniport → NIC Driver (커널)

9. 사용 시나리오

DPDK를 선택해야 할 때

  • 초저지연이 필수인 금융 트레이딩, 실시간 제어
  • 고처리량 패킷 처리 (10Gbps 회선 포화)
  • 네트워크 장비 (방화벽, 로드밸런서) 구현
  • 예측 가능한 지터가 필요한 산업 제어
  • 패킷 헤더 레벨의 세밀한 제어 필요
  • 전용 서버에서 NIC를 독점 사용 가능한 환경

Socket을 선택해야 할 때

  • 빠른 프로토타이핑과 개발 편의성 우선
  • OS 네트워크 기능 (방화벽, QoS, 라우팅) 활용
  • 다른 앱과 NIC 공유가 필요한 환경
  • 특별한 드라이버/권한 없이 동작해야 할 때
  • CPU 효율이 중요한 저트래픽 환경
  • 표준 프로토콜 호환성 (TCP, DNS, DHCP 등)

이 테스트 도구의 활용법

동일한 테스트 시나리오를 두 탭에서 실행하여 비교하세요:
  1. 성능 모니터 탭에서 DPDK/Socket PPS, Mbps를 나란히 확인
  2. Req/Resp 모드로 RTT 비교 — DPDK는 μs, Socket은 ms 단위
  3. TX 모드로 최대 처리량 비교 — PPS 제한 없이 전송
  4. CSV 로그 파일로 시계열 분석 (PacketLogger가 TX/RX 분리 기록)
