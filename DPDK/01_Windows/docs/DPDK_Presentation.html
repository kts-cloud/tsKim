1.2 DPDK vs Socket 비교

항목DPDK (Kernel Bypass)Socket (커널 경유)
패킷 처리사용자 공간에서 직접 NIC 접근커널 네트워크 스택 경유
컨텍스트 스위치없음 (poll-mode)매 syscall마다 발생
메모리 복사Zero-copy (DMA → hugepage)커널 ↔ 사용자 공간 복사
인터럽트없음 (CPU polling)하드웨어 인터럽트 기반
RTT (1패킷)~0.1ms~0.5-1ms
Throughput수백만 PPS 가능~100K PPS 수준
ARP/라우팅직접 구현 필요OS가 자동 처리

1.3 데이터 흐름

TX 데이터 흐름 (DPDK):
  C# 패킷 구성 → Marshal.Copy → mbuf (hugepage) → rte_eth_tx_burst → NIC DMA → Wire

RX 데이터 흐름 (DPDK):
  Wire → NIC DMA → mbuf (hugepage) → rte_eth_rx_burst → Marshal.Copy → C# 처리

Req/Resp 데이터 흐름 (최적화):
  C# 템플릿 → shim_reqresp_once() [네이티브 루프: TX→Poll RX→RTT측정] → 결과 반환

4. Windows DLL Import 버그 수정

DPDK를 Windows에서 공유 라이브러리(DLL)로 빌드할 때 발생하는 3가지 종류의 버그를 발견하고 수정했습니다. 이 버그들은 Linux에서는 발생하지 않으며, Windows DLL 메커니즘의 차이에서 기인합니다.

4.1 버그 유형 1: Global Data 변수

증상

다른 DLL에 정의된 전역 변수(extern)를 읽을 때 쓰레기 값을 반환

원인

// rte_mempool_ops.c 에서 정의:
struct rte_mempool_ops_table rte_mempool_ops_table;  // DLL A

// rte_mempool.h 에서 참조 (DLL B):
extern struct rte_mempool_ops_table rte_mempool_ops_table;
// Windows: __declspec(dllimport) 없으면 JMP thunk 주소를 데이터로 읽음!
Linux:   extern 변수 → PLT/GOT → 실제 데이터 주소 (투명)
Windows: extern 변수 → JMP thunk (FF 25 xx xx) → 6바이트 명령어를 데이터로 해석 = 쓰레기값
         __declspec(dllimport) → IAT에서 직접 실제 주소 로드 (정상)

수정

파일변수수정 내용
lib/mempool/rte_mempool.hrte_mempool_ops_table__declspec(dllimport) 추가
lib/ethdev/rte_ethdev_core.hrte_eth_fp_ops__declspec(dllimport) 추가
lib/eal/include/generic/rte_cycles.hrte_delay_us, eal_timer_source__declspec(dllimport) 추가

4.2 버그 유형 2: TLS (__thread) 변수

증상

rte_lcore_id()가 항상 0을 반환 (모든 스레드가 lcore 0으로 인식)

원인

// rte_lcore.h:
RTE_DECLARE_PER_LCORE(unsigned, _lcore_id);
// 내부적으로: __thread unsigned per_lcore__lcore_id;

// 문제: Windows TLS는 DLL 경계를 넘을 수 없음
// DLL A의 __thread 변수를 DLL B에서 접근하면 다른 _tls_index 사용

수정

TLS 변수 대신 accessor 함수 패턴 사용:

// eal_common_thread.c — 같은 DLL 내에서 accessor 함수 추가:
unsigned int rte_lcore_id_get(void) {
    return RTE_PER_LCORE(_lcore_id);
}

// rte_lcore.h — 외부 DLL에서는 accessor 사용:
static inline unsigned rte_lcore_id(void) {
    return rte_lcore_id_get();  // DLL 경계를 함수 호출로 넘김
}

4.3 버그 유형 3: Global Function Pointer

증상

rte_delay_us() 호출 시 잘못된 주소로 점프하여 크래시

원인

전역 함수 포인터(void (*rte_delay_us)(unsigned int))도 global data와 동일한 JMP thunk 문제 발생

Cleanup — 순서적 정리

public void Cleanup()
{
    // 1. Ethernet 장치 중지 및 닫기 (포트 설정 완료된 경우만)
    if (_portSetupDone) {
        rte_eth_dev_stop(PortId);
        rte_eth_dev_close(PortId);
    }
    // 2. EAL 정리 (hugepage 메모리 해제)
    if (_ealInitDone)
        rte_eal_cleanup();
    // 3. 관리 메모리(argv 포인터) 해제
    Marshal.FreeHGlobal(_argvPtr);
}
DPDK 재초기화 불가: rte_eal_init()은 프로세스 당 1회만 호출 가능. 전역 상태(mem_config, hugepage 매핑, TAILQ, virt2phys 핸들)가 cleanup 후에도 잔존. 재초기화가 필요하면 프로세스 재시작 방식 사용 (SettingsTab → 재시작 버튼).

C# Core/DpdkInterop.cs

C# ↔ dpdk_shim.dll 간의 모든 함수 호출을 정의합니다. 모든 선언은 CallingConvention.Cdecl.

API 분류

그룹함수SEH 보호용도
EALshim_eal_init, shim_eal_cleanup, shim_versionO초기화/정리

6.4 shim_reqresp_once() — 단일 패킷 RTT 측정

가장 성능이 중요한 함수입니다. 전체 TX→RX→RTT 계산을 네이티브에서 수행하여 P/Invoke 오버헤드를 제거합니다.

DPDK_API int shim_reqresp_once(
    uint16_t port_id, struct rte_mempool *pool,
    const uint8_t *template_pkt, uint16_t pkt_len, uint16_t packet_id,
    uint32_t expected_src_ip, uint16_t expected_dst_port,
    int timeout_ms,
    uint8_t *resp_buf, uint16_t resp_buf_size,
    shim_reqresp_result_t *result,
    const uint8_t *local_mac, uint32_t local_ip)
{
    // 1. mbuf 할당 + 템플릿 복사
    struct rte_mbuf *mbuf = rte_pktmbuf_alloc(pool);
    memcpy(data, template_pkt, pkt_len);

    // 2. ★ 증분 IP 체크섬 (RFC 1624)
    //    PacketId만 변경되므로 전체 20바이트 재계산 대신 4바이트 산술만 수행
    uint16_t old_id, new_id = rte_cpu_to_be_16(packet_id);
    memcpy(&old_id, data + ETH_HDR_LEN + 4, 2);
    // ~new_ck = ~old_ck + ~old_id + new_id
    uint32_t sum = (uint16_t)~old_ck + (uint16_t)~old_id + new_id;
    sum = (sum >> 16) + (sum & 0xffff); sum += (sum >> 16);
    uint16_t new_ck = (uint16_t)~sum;

    // 3. TX Burst
    rte_eth_tx_burst(port_id, 0, tx_pkts, 1);
    QueryPerformanceCounter(&send_tick);  // 전송 시각 기록

    // 4. ★ RX 폴링 루프 (rte_pause + 타임아웃 체크)
    while (1) {
        nb_rx = rte_eth_rx_burst(port_id, 0, rx_pkts, 32);

        if (nb_rx == 0) {
            rte_pause();  // x86 PAUSE 명령 (CPU 양보 힌트)
            if (++empty_polls < 64) continue;
            // 64번 연속 빈 폴 → 타임아웃 체크
        }

        for (각 수신 패킷) {
            // ARP 요청 → send_arp_reply() (인라인 ARP 프록시)
            // UDP 매칭: src_ip == expected && dst_port == expected
            if (매칭 성공) {
                QueryPerformanceCounter(&now_tick);
                result->rtt_ms = (now - send) * 1000.0 / freq;
                return 0;  // 성공
            }
        }

        // 타임아웃 체크
        if (elapsed >= timeout_sec) break;
    }
    result->status = 1;  // 타임아웃
    return 1;
}

{
    struct rte_mbuf *m = rte_pktmbuf_alloc(g_pool);
    if (!m) return ERR_MEM;

    uint16_t total = 0;
    for (struct pbuf *q = p; q; q = q->next) {
        rte_memcpy(rte_pktmbuf_mtod_offset(m, void*, total),
                   q->payload, q->len);

lwIP의 raw TCP API (tcp_new, tcp_connect, tcp_write, tcp_recv 콜백)를 사용하여 FTP 프로토콜을 구현합니다.

상태 전이 다이어그램

IDLE → CONNECTING → WAIT_WELCOME(220) → SEND_USER → WAIT_USER(331)
  → SEND_PASS → WAIT_PASS(230) → READY
  → SEND_PASV → WAIT_PASV(227) → DATA_CONNECTING → SENDING_CMD
  → TRANSFERRING → WAIT_DONE(226) → READY
  → ERROR (어디서든 전이 가능)

PASV 모드

FTP 데이터 전송은 PASV (Passive) 모드를 사용합니다. 서버의 227 응답에서 데이터 채널의 IP:포트를 파싱합니다:

// 227 Entering Passive Mode (h1,h2,h3,h4,p1,p2)
// → data_ip   = h1.h2.h3.h4
// → data_port = p1 * 256 + p2
static int ftp_parse_pasv(const char *resp, ip_addr_t *ip, uint16_t *port)
{
    int h1,h2,h3,h4,p1,p2;
    const char *s = strchr(resp, '(');
    if (!s) return -1;
    if (sscanf(s, "(%d,%d,%d,%d,%d,%d)", &h1,&h2,&h3,&h4,&p1,&p2) != 6)
        return -1;
    IP4_ADDR(ip, h1, h2, h3, h4);
    *port = (uint16_t)(p1 * 256 + p2);
    return 0;
}

동기 래퍼 패턴: ftp_poll_wait()

lwIP는 NO_SYS=1 모드에서 비동기 콜백 기반으로 동작합니다. FTP 명령의 응답을 기다리기 위해 busy-loop 폴링을 수행합니다:

static int ftp_poll_wait(int target_state, uint32_t timeout_ms)
{
    uint32_t start = sys_now();
    while (g_ftp.state != target_state &&
           g_ftp.state != FTP_STATE_ERROR)
    {
        dpdk_lwip_poll_once();      // rx_burst → lwIP 입력
        sys_check_timeouts();       // lwIP 타이머 처리 (TCP 재전송 등)

        if (sys_now() - start > timeout_ms)
            return -1;              // 타임아웃
    }
    return (g_ftp.state == target_state) ? 0 : -1;
}

10.1 GC 압력 제거

문제

RepeatCount=0(무한 반복) vs RepeatCount=N(N회 반복)에서 평균 RTT 차이 발생

RepeatCount=0:
  ResultQueue → 10,000개 도달 → enqueue 중단 → 힙 할당 0 → GC 없음 ✓

RepeatCount=N:
  UI Timer가 큐 drain → 큐 < 10,000 유지 → 매 패킷 4~5개 힙 할당
  → new ReqRespResult, Encoding.ASCII.GetString, TrimEnd, UintToIp
  → ~2,000 alloc/sec → GC 간섭 → RTT 증가 ✗

해결

// Before: 성공 시에도 결과를 큐에 enqueue (힙 할당 4~5개/패킷)
if (ResultQueue.Count < 10000)
    ResultQueue.Enqueue(new ReqRespResult { ... });  // ← GC 압력

// After: 성공 시 카운터만 업데이트 (힙 할당 0개)
if (nativeResult.Status == 0) {
    Stats.AddReceived(nativeResult.RttMs);  // ← Interlocked만 사용
}
// 타임아웃 시에만 enqueue (드문 이벤트)

Q. 포트가 0개로 감지됨

  • NIC가 netuio에 바인딩되었는지 확인: toggle_netuio.ps1 -Mode Bind
  • 장치 관리자에서 NIC가 "netuio"로 표시되는지 확인
  • DPDK DLL이 실행 디렉토리에 복사되었는지 확인: copy_dlls.ps1

Q. 링크 UP이 안 됨

  • 물리적 케이블 연결 확인
  • SFP+ 모듈 호환성 확인 (Intel X550은 Intel SFP+ 권장)
  • Link Speed 설정을 "Auto (협상)"으로 변경

Q. DPDK 재초기화가 안 됨

  • DPDK EAL은 프로세스 당 1회만 초기화 가능 (아키텍처 제한)
  • "재시작" 버튼으로 앱을 재시작하세요 (설정 자동 저장됨)

Q. Req/Resp 첫 몇 패킷이 느림

  • NIC/DMA 워밍업이 자동 수행됩니다 (64 더미 패킷)
  • Gratuitous ARP 3회 전송으로 상대 ARP 캐시 사전 충전
  • 상대 장치가 ARP를 느리게 학습하면 TX 모드를 몇 초 실행 후 전환

Q. TestSigning 설정이 안 됨

  • BIOS에서 Secure Boot가 꺼져 있는지 확인
  • PowerShell 관리자 모드에서: bcdedit /set testsigning on
  • 재부팅 필요

Q. DLL 로드 실패

  • copy_dlls.ps1로 모든 DPDK DLL을 실행 디렉토리에 복사
  • dpdk_shim.dll이 실행 디렉토리에 있는지 확인
  • Visual C++ Redistributable 설치 확인
