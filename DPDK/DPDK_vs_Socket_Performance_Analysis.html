Performance Analysis

DPDK vs Socket 성능 분석 tcon.read/write RTT 차이 없는 원인 — PG Board HW 병목 분석

DPDK kernel bypass와 기존 Socket 방식의 tcon.read/write RTT를 비교한 결과, 평균 32μs (2.0%) 차이로 사실상 동일한 성능을 보임. 이 문서는 그 원인을 RTT 구성 요소별로 분석합니다.

Section 03

RTT 구성 요소 분석

tcon.read/write 전체 RTT 구성: [C# App] ──TX──> [NIC] ══════> [PG Board] ══════> [NIC] ──RX──> [C# App] ~100μs ~5μs ~1500μs (HW처리) ~5μs ~100μs ├── SW Send ──┤├── Network ─┤├──── PG HW ────┤├── Network ─┤├── SW RX ──┤ ~100μs ~5μs ~1500μs ~5μs ~100μs ─────────────────────────────────────────────────────────────────── 총 RTT ≈ 1710μs SW 제어 가능: ~200μs (12%) PG HW 고정: ~1500μs (88%) ← 병목
핵심: 전체 RTT의 88%가 PG Board HW 처리 시간이며, DPDK/Socket이 제어할 수 있는 구간은 12%에 불과합니다.

PG HW 처리 시간 산출 근거

PG HW 시간은 PG 보드에서 직접 측정한 값이 아니라, hw_reqresp_once_mc의 네이티브 RTT에서 네트워크 전파 지연을 뺀 간접 측정값입니다.

hw_reqresp_once_mc 내부 측정 지점: rte_eth_tx_burst() rte_eth_rx_burst() → IP/Port 매칭 ↓ ↓ QueryPerformanceCounter(&send_tick) QueryPerformanceCounter(&now_tick) │ │ └──── 네이티브 RTT = now - send ─────────┘ 네이티브 RTT 구성: ├─ NIC TX wire delay ~3-5μs (1Gbps 직결, NIC datasheet) ├─ PG Board HW 처리 ~1510μs ← 나머지 전부 └─ NIC RX wire delay ~3-5μs (1Gbps 직결, NIC datasheet) 로그 예시: [RX] 8001<8001: tcon.read ... RET:OK [1523μs] PG HW ≈ 1523 - 5 - 5 = ~1513μs
측정 정밀도: QueryPerformanceCounter는 Windows 고정밀 타이머(~0.1μs 해상도). tx_burst 직후부터 rx_burst 매칭까지의 경과 시간이므로, C# 오버헤드를 포함하지 않는 순수 "NIC 밖 왕복 시간"입니다. Network wire delay(~10μs)는 1Gbps 직결 + 케이블 수 미터 환경에서 고정값이므로, 나머지는 PG 보드의 TCON 레지스터 읽기/쓰기 + 응답 패킷 생성 시간입니다.
Section 04

DPDK vs Socket Send 경로 비교

Socket Send Path (~25-50μs)

Kernel Syscall 기반

  • Encoding.ASCII.GetBytes — ~5μs
  • new IPEndPoint — ~3μs
  • UdpClient.Send (kernel syscall) — ~15-30μs

총: ~25-50μs

DPDK Send Path (~80-110μs)

Userspace DMA 기반

  • BuildUdpPacket (new byte[], header) — ~35-50μs
  • AllocMbuf (DPDK mempool) — ~10-15μs
  • Marshal.Copy (managed -> native) — ~15-20μs
  • TxBurst (DMA) — ~3-5μs

총: ~80-110μs

Send 경로 오버헤드 비교: Socket: GetBytes(5) + IPEndPoint(3) + Syscall(15-30) = ~25-50μs DPDK: BuildPkt(35-50) + Mbuf(10-15) + Marshal(15-20) + DMA(3-5) = ~80-110μs ─────── DPDK 패킷 구성 오버헤드: +50-60μs (byte[] 할당, 헤더 구축, mbuf lifecycle) DPDK NIC-level TX 이득: -10-25μs (DMA vs kernel syscall) ─────────────────────────────────────────────────────────────────── Net: DPDK가 Send에서 ~30-40μs 더 느림
Section 05

RX 경로 비교

[v1.1 업데이트] DPDK 동기 명령(tcon.read/write 등 waitMs > 0)은 hw_reqresp_once_mc 네이티브 경로를 사용하여 TX→RX→매칭을 C 코드 내에서 처리합니다. Socket 경로만 C# 파싱을 사용합니다.

Socket RX Path (~200-300μs)

C# Managed 파싱

  • ReceiveAsync (kernel → C#) — ~20-40μs
  • ProcessRxPacket string 파싱 — ~80-100μs
  • ManualResetEventSlim signal — ~50-100μs
  • CheckCmdAck spin-wait 감지 — ~50-100μs

총: ~200-300μs

DPDK RX Path (~10-30μs)

Native C 처리 (hw_reqresp_once_mc)

  • hw_reqresp_once_mc — TX+RX+매칭을 native C에서 처리
  • rte_eth_rx_burst polling — ~3-5μs
  • IP/port 매칭 + payload 추출 — ~5-10μs
  • C#에 RespLen + RttMs 반환 — ~2-5μs

총: ~10-30μs

RX 경로 (동기 명령 — tcon.read/write): Socket: ReceiveAsync(20-40) + C# 파싱(80-100) + Signal(50-100) + SpinWait(50-100) = ~200-300μs DPDK: hw_reqresp_once_mc native 처리 (RxBurst + IP/port 매칭 + payload 추출) = ~10-30μs ─────────────────────────────────────────────────────────────────── DPDK RX 이득: -170~270μs (C# 파싱 + 스레드 동기화 완전 제거) 참고: 비동기 명령 (pg.status 등 waitMs=0)은 양쪽 모두 C# RxPollLoop/ReceiveLoop 사용
v1.1 핵심: DPDK 동기 명령의 RX 경로가 hw_reqresp_once_mc로 변경되어 C# 파싱 오버헤드가 완전히 제거되었습니다. 이로 인해 DPDK RX가 ~200μs 빠릅니다. 하지만 TX에서 ~50μs 느리므로 net 이득은 ~150μs이며, PG HW 처리 시간(~1400μs) 대비 ~9% 개선입니다.
Section 05-1

hw_reqresp_once_mc 내부 동작 상세

hw_reqresp_once_mc는 하나의 native C 함수 호출로 "패킷 전송 → NIC 폴링 → 매칭 → payload 추출"을 전부 처리합니다. C#으로 반환될 때는 응답 문자열과 RTT가 이미 완성되어 있습니다.

C# CommPgDriver.Dp860SendCmd │ ├─ BuildUdpPacket() C#에서 Ethernet/IP/UDP 헤더 + payload 조립 │ 결과: byte[] templatePkt (완성된 raw 패킷) │ ├─ ReqRespOnceMc(templatePkt, expectedSrcIp, expectedDstPort, timeout) │ │ │ └─── [여기서부터 전부 native C — hwio.dll] ────────────── │ │ │ ├─ Step 1 rte_pktmbuf_alloc mbuf 할당 (hugepage DMA) │ ├─ Step 2 rte_pktmbuf_append mbuf 공간 확보 │ ├─ Step 3 memcpy(data, template) 패킷 복사 │ ├─ Step 4 PacketId 교체 + IP checksum 증분 갱신 │ ├─ Step 5 rte_eth_tx_burst ★ NIC DMA 전송 │ ├─ Step 6 QueryPerformanceCounter TX 시각 기록 │ │ │ ├─ Step 7 RX 폴링 루프 ────────────────────────── │ │ while (timeout 안 됨) { │ │ rte_eth_rx_burst(32개씩) ★ NIC에서 수신 │ │ for (each 수신 패킷) { │ │ ARP? → send_arp_reply() 후 free │ │ IPv4+UDP? → SrcIP == expected? DstPort == expected? │ │ 매칭 성공 → RTT 계산 + memcpy(payload) → return 0 │ │ 매칭 실패 → spill_ring에 보관 ★ │ │ } │ │ } │ │ return 1 (타임아웃) │ │ │ ─────[native C 종료]────────────────────────────── │ ├─ result: { status=0, resp_buf="tcon.read...RET:OK", rtt_ms=1.52 } └─ C# 결과 처리 (문자열 파싱 최소화)

왜 Native가 C#보다 빠른가? — 3가지 핵심 이유

1

스레드 동기화 완전 제거 (~100-150μs 이득)

Socket / C# 경로

TX 스레드 → 패킷 전송
    ← PG 응답 도착 (NIC)
RX 스레드 → 패킷 수신 + 파싱
RX 스레드 → ManualResetEventSlim.Set()
TX 스레드 → SpinWait 루프에서 감지 ★
TX 스레드 → 결과 처리
비용: OS 스레드 스케줄러 + CPU 캐시 이동
→

DPDK Native 경로

tx_burst() → 전송
while { rx_burst() → 수신 }
if (매칭) → return ★
 
 
 
비용: 0 (동일 함수, 동일 스레드)
핵심: TX와 RX가 다른 스레드면 "응답 왔다"는 신호를 전달하기 위해 ManualResetEventSlim + SpinWait가 필요합니다. 이 과정에서 OS 스레드 스케줄러 개입과 CPU L1/L2 캐시 라인의 코어 간 이동이 발생합니다. Native에서는 TX→폴링→매칭→반환이 하나의 함수 스택 프레임에서 끝나므로 동기화 비용이 완전히 0입니다.
2

Managed 메모리 할당 · GC 압력 제거 (~30-50μs 이득)

C# 경로 — GC 대상 할당

매 패킷마다 3~4회 힙 할당

var payloadBytes = new byte[udpPayloadLen]; // ★ GC 할당 #1 Marshal.Copy(dataPtr, payloadBytes, 0, len); var sAnsiData = Encoding.ASCII.GetString(payloadBytes); // ★ GC 할당 #2 var sCmdAck = sData[..(posAck + retXxLen)].Trim(); // ★ GC 할당 #3 sCmdAck = sCmdAck.Replace("\n", "").Replace("\r\r", "\r"); // ★ GC 할당 #4

.NET GC Gen0 컬렉션 시 수십 μs 스파이크 발생 가능

Native C — GC 없음

힙 할당 0회

memcpy(&rx_src_ip, rx_data + 26, 4); // 스택 변수, 4바이트 if (rx_src_ip != expected_src_ip) // 정수 비교 goto mc_spill_pkt; memcpy(&rx_dst_port, rx_data + 36, 2); // 스택 변수, 2바이트 if (rx_dst_port != expected_dst_port) // 정수 비교 goto mc_spill_pkt; // 매칭 성공 시에만 1회 복사 memcpy(resp_buf, rx_data + payload_off, payload_len);

모든 변수가 스택/레지스터 → CPU 캐시에서 직접 처리

핵심: C# 경로는 매 패킷마다 byte[], string 등 managed 힙 객체를 3~4개 할당합니다. 이는 .NET GC의 추적 대상이 되며, Gen0 컬렉션이 트리거되면 수십 μs STW(Stop-The-World) 일시정지가 발생합니다. Native C는 스택 변수와 memcpy만 사용하여 GC 자체가 존재하지 않습니다.
3

패킷 식별: 문자열 파싱 → 바이너리 헤더 정수 비교 (~50-80μs 이득)

주의: RET:OK / RET:NG 최종 판정은 양쪽 모두 C#에서 수행합니다. Native가 제거하는 것은 "이 패킷이 내 것인가?"를 판별하는 패킷 식별 단계입니다.

Socket과의 차이: Socket 모드에서는 OS 커널 UDP 스택이 포트 기반으로 패킷을 자동 분배합니다 (UdpClient.bind(port 8001) → 해당 포트 패킷만 수신). DPDK는 커널을 우회하므로 NIC의 모든 패킷(ARP, mDNS, 모든 CH)이 애플리케이션에 직접 도착하며, 패킷 식별을 직접 수행해야 합니다.

Socket 모드: [NIC] → [OS 커널 UDP 스택] → 포트별 자동 분배 (비용 0μs) │ ├─ port 8000 → baseUdpClient (ConnCheck) ├─ port 8001 → pgUdpClients[0] (CH0 전용) ← 내 것만 도착 ├─ port 8002 → pgUdpClients[1] (CH1 전용) └─ port 5353 (mDNS) → 커널이 걸러줌 (C#에 안 옴) DPDK 모드: [NIC] → [DPDK userspace] → 모든 패킷이 한 곳으로 │ └─ rte_eth_rx_burst → 전부 직접 분류해야 함 ├─ ARP (0x0806) → ARP 처리 ├─ mDNS (port 5353) → spill / 버림 ├─ CH1 응답 (port 8002) → spill ring └─ CH0 응답 (port 8001) → ★ 매칭!
Native C (hw_reqresp_once_mc) — 패킷 식별: "이 패킷이 내 것인가?" → IP(4B) + Port(2B) 정수 비교 매칭 성공 → resp_buf에 raw payload 복사 ("tcon.read 0x1234\rRET:OK") C# (CommPgDriver.Dp860SendCmd) — 결과 판정: resp.Contains("RET:OK") → OK resp.Contains("RET:NG") → NG resp.Contains("RET:INFO") → 중간 응답 (후속 대기)
Socket C# — 패킷 식별 + 파싱 전부

매 수신 패킷마다 O(n) 문자열 처리

// ① 매 패킷: byte[] → string 전체 변환 — O(n) var sAnsiData = Encoding.ASCII.GetString(payloadBytes); // ② 이전 잔여분 결합 (불완전 패킷 처리) var sData = txRxData.RxPrevStr + sAnsiData; // ③ "RET:" 위치 탐색 — O(n) 바이트 스캔 var posAck = sData.IndexOf("RET:", ...); // ④ 부분 문자열 추출 + 제어 문자 치환 — O(n) × 3회 var sCmdAck = sData[..(posAck + retXxLen)].Trim(); sCmdAck = sCmdAck.Replace("\n", "").Replace("\r\r", "\r"); // ⑤ 비매칭 패킷도 여기까지 전부 파싱됨 (버려지더라도)

모든 수신 패킷(mDNS, 다른 CH 등)에 대해 실행 = ~80-100μs/패킷

DPDK — Native 식별 + C# 판정

Native: O(1) 정수 비교 → C#: Contains 1회

// ① IP/Port 정수 비교로 "내 것인가" 식별 — O(1) memcpy(&rx_src_ip, rx_data + 26, 4); if (rx_src_ip != expected_src_ip) goto mc_spill_pkt; memcpy(&rx_dst_port, rx_data + 36, 2); if (rx_dst_port != expected_dst_port) goto mc_spill_pkt; // ② 매칭 성공 → payload만 복사 memcpy(resp_buf, rx_data + payload_off, payload_len);
// ③ Native가 반환한 완성된 응답만 판정 — 1회 string response = Encoding.ASCII.GetString(t_respBuf, 0, result.RespLen); if (resp.Contains("RET:OK", StringComparison.OrdinalIgnoreCase)) result = WAIT_OBJECT_0;

Native: 비매칭 패킷은 6바이트 비교 후 즉시 spill → C#: 매칭된 1개만 처리

멀티채널 안전: Spill Ring

문제: 4CH 멀티채널에서 CH0이 hw_reqresp_once_mc로 대기 중일 때 CH1/2/3의 응답이 먼저 NIC에 도착하면? [NIC RX Queue] │ ├─ CH1 응답 (SrcIP=.12) → 매칭 실패 → spill_ring에 보관 ★ 버리지 않음 ├─ CH3 응답 (SrcIP=.14) → 매칭 실패 → spill_ring에 보관 ├─ CH0 응답 (SrcIP=.11) → 매칭 성공! → RTT 계산 + return 0 │ └─ 나머지 패킷도 spill_ring에 보관 ─────────────────────────────────────────────────────────────────── 이후 C# RxPollLoop → hw_dispatch_poll(): 1. spill_ring에서 먼저 dequeue (CH1/CH3 응답) 2. NIC에서 추가 burst → CH1/CH3 응답이 C# ProcessRxPacket으로 정상 전달 결과: 어떤 채널의 패킷도 유실되지 않음
spill_ring: Lock-free SPSC(Single-Producer Single-Consumer) ring으로 구현. hw_reqresp_once_mc가 enqueue(SP), hw_dispatch_poll이 dequeue(SC). 락 없이 채널 간 패킷을 안전하게 중계합니다.

RX 폴링 중 ARP 즉시 응답

// RX 폴링 루프 내부 if (ether_type == arp_ethertype) { send_arp_reply(port_id, pool, rx_data, rx_len, local_mac); // ★ native에서 즉시 응답 goto mc_free_pkt; // C#으로 올리지 않음 }

RX 폴링 중 ARP 요청이 도착하면 C#으로 올리지 않고 native에서 즉시 ARP Reply를 전송합니다. 이는 PG 보드가 우리 MAC을 물어올 때 응답 지연 없이 처리되어 통신 안정성을 높입니다.

Section 06

왜 DPDK가 빠르지 않은가?

┌─────────────────────────────────────────────────────────┐ │ tcon.read/write RTT ≈ 1600μs │ ├────────┬────────────────────────────┬───────────────────┤ │ SW TX │ PG Board HW 처리 │ SW RX │ ├────────┼────────────────────────────┼───────────────────┤ DPDK: │ 80-110 │ 동일 │ 10-30 │ Socket: │ 25-50 │ 동일 │ 200-300 │ ├────────┼────────────────────────────┼───────────────────┤ 합계: │ │ │ │ DPDK: │ +50μs │ 0 │ -200μs │ net: -150μs └────────┴────────────────────────────┴───────────────────┘ ↑ ↑ TX: C# 패킷 구성 오버헤드 RX: hw_reqresp_once_mc (BuildUdpPacket+Marshal) (C# 파싱 완전 제거)
v1.1 업데이트: hw_reqresp_once_mc 적용 후, DPDK RX가 ~200μs 빨라졌습니다. TX에서 ~50μs 느리지만 RX에서 ~200μs 이득이므로 net ~150μs 개선. 다만 PG HW 처리(~1400μs)가 여전히 지배적이므로 전체 RTT 대비 ~9% 수준입니다. 실측 평균 차이 32μs(2%)와의 괴리는 PG HW 처리 시간의 변동폭(±100μs)에 묻히기 때문입니다.

최적화 적용 결과

#최적화대상효과상태
1 ApplyTconWriteDelay 제거 TX-to-TX gap 코드 정리 (설정값 0) 적용
2 ToUpperInvariant → OrdinalIgnoreCase RX 파싱 ~20-30μs 감소 적용
3 SpinWait(20) → SpinWait(4) CheckCmdAck ~10-20μs 감소 적용

7 ARP ResolveDestMac 에러 로깅 개선 DPDK 진단 AllocMbuf/AppendMbuf NULL, AccessViolation 개별 로깅 적용

항목 2, 3은 DPDK/Socket 양쪽 모두에 적용. 항목 5(hw_reqresp_once_mc)는 DPDK 전용 최적화로, 동기 명령의 RX 경로에서 C# 파싱·스레드 동기화를 완전 제거하여 ~200μs 이득. 항목 6은 프로세스 비정상 종료 시 DPDK shared config 오염으로 재시작 후 mbuf 할당 불가 문제를 해결합니다.

Section 08

결론 및 권장사항

  • 1
    DPDK의 실제 이점: hw_reqresp_once_mc로 RX 경로가 네이티브 C로 전환되어 이론적 ~150μs 이득. 히스토그램 분포에서 DPDK가 1000-1500μs 구간에 집중. Median ~109μs, P99 ~93μs 개선 확인.
  • 2
    DPDK가 빛나는 상황: 대량 패킷 burst, FTP 전송(DpdkFtpEngine), 멀티채널 동시 통신에서 kernel bypass 효과 극대화. 단건 request/response에서도 tail latency(P95/P99) 개선 효과.
  • 3
    tcon 단건 통신에서는: PG Board HW 처리가 전체 RTT의 88%를 차지. DPDK net 이득 ~150μs이나 PG HW 변동폭(±100μs)에 묻혀 실측 평균 차이는 32μs(2.0%).
  • 4
    안정성 개선: --no-shconf 추가로 프로세스 강제종료 후 hugepage 오염 방지. ARP 에러 로깅 개선으로 현장 진단 용이.
  • 5
    권장: DPDK 유지 (FTP + 멀티채널 + tail latency 이점 + 안정성 개선). tcon RTT 근본 개선은 PG Board 펌웨어 최적화 필요.
