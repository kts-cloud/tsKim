Section 03

RTT 구성 요소 분석

tcon.read/write 전체 RTT 구성: [C# App] ──TX──> [NIC] ══════> [PG Board] ══════> [NIC] ──RX──> [C# App] ~100μs ~5μs ~1500μs (HW처리) ~5μs ~100μs ├── SW Send ──┤├── Network ─┤├──── PG HW ────┤├── Network ─┤├── SW RX ──┤ ~100μs ~5μs ~1500μs ~5μs ~100μs ─────────────────────────────────────────────────────────────────── 총 RTT ≈ 1710μs SW 제어 가능: ~200μs (12%) PG HW 고정: ~1500μs (88%) ← 병목
핵심: 전체 RTT의 88%가 PG Board HW 처리 시간이며, DPDK/Socket이 제어할 수 있는 구간은 12%에 불과합니다.
Section 04

DPDK vs Socket Send 경로 비교

Socket Send Path (~25-50μs)

Kernel Syscall 기반

  • Encoding.ASCII.GetBytes — ~5μs
  • new IPEndPoint — ~3μs
  • UdpClient.Send (kernel syscall) — ~15-30μs

총: ~25-50μs

DPDK Send Path (~80-110μs)

Userspace DMA 기반

  • BuildUdpPacket (new byte[], header) — ~35-50μs
  • AllocMbuf (DPDK mempool) — ~10-15μs
  • Marshal.Copy (managed -> native) — ~15-20μs
  • TxBurst (DMA) — ~3-5μs

총: ~80-110μs

Send 경로 오버헤드 비교: Socket: GetBytes(5) + IPEndPoint(3) + Syscall(15-30) = ~25-50μs DPDK: BuildPkt(35-50) + Mbuf(10-15) + Marshal(15-20) + DMA(3-5) = ~80-110μs ─────── DPDK 패킷 구성 오버헤드: +50-60μs (byte[] 할당, 헤더 구축, mbuf lifecycle) DPDK NIC-level TX 이득: -10-25μs (DMA vs kernel syscall) ─────────────────────────────────────────────────────────────────── Net: DPDK가 Send에서 ~30-40μs 더 느림
Section 05

RX 경로 비교

DPDK와 Socket 모두 동일한 C# 파싱 로직을 사용하므로, RX 경로의 오버헤드는 사실상 동일합니다.

RX 처리 경로 (DPDK / Socket 공통): 1. ProcessRxPacket string 파싱 ~80-100μs ToUpperInvariant, IndexOf, Substring 등 2. ManualResetEventSlim signal ~50-100μs RX 스레드 → CheckCmdAck 대기 스레드 깨움 3. CheckCmdAck spin-wait 감지 ~50-100μs SpinWait loop에서 응답 감지 ─────────────────────────────────────────────────────────────────── 총 RX 오버헤드: ~200-300μs (DPDK/Socket 동일)
결론: RX 경로는 100% C# managed code이므로, 네트워크 레이어(DPDK/Socket)의 차이가 반영되지 않습니다. 두 방식 모두 ~200-300μs의 동일한 오버헤드를 가집니다.
Section 06

왜 DPDK가 빠르지 않은가?

┌─────────────────────────────────────────────────────────┐ │ tcon.read/write RTT ≈ 1600μs │ ├────────┬────────────────────────────┬───────────────────┤ │ SW TX │ PG Board HW 처리 │ SW RX │ │ ~100μs │ ~1400μs (88%) │ ~100μs │ ├────────┼────────────────────────────┼───────────────────┤ DPDK: │ 80-110 │ 동일 │ 200-300 │ Socket: │ 25-50 │ 동일 │ 200-300 │ └────────┴────────────────────────────┴───────────────────┘ ↑ DPDK가 여기서 50μs 더 느림 하지만 전체 1600μs 중 3%에 불과
핵심 인사이트: PG 보드 HW 처리 시간(~1400μs)이 전체 RTT의 88%를 차지하므로, 네트워크 레이어 최적화의 효과가 미미합니다. DPDK의 kernel bypass는 NIC TX에서 ~10-25μs를 절약하지만, C# 패킷 구성 오버헤드(BuildUdpPacket + Marshal.Copy)가 ~50-60μs를 추가하여 오히려 Send 경로가 느려집니다.

최적화 항목 2, 3은 DPDK/Socket 양쪽 모두에 적용되어 공통적으로 ~30-50μs 감소 효과를 가져왔으나, 두 방식 간 상대적 차이에는 영향이 없습니다.

Section 08

결론 및 권장사항

  • 1
    DPDK의 실제 이점: NIC RTT는 동일하지만, 히스토그램 분포에서 DPDK가 1000-1500μs 구간에 더 많은 패킷이 집중됩니다. Median에서 ~109μs 차이, P99 tail latency에서 ~93μs 개선이 확인되었습니다.
  • 2
    DPDK가 빛나는 상황: 대량 패킷 burst, FTP 전송, 멀티채널 동시 통신 시나리오에서 kernel bypass 효과가 극대화됩니다. 단건 request/response에서는 이점이 제한적입니다.
  • 3
    tcon 단건 통신에서는: PG Board HW 처리가 전체 RTT의 88%를 차지하므로, DPDK와 Socket의 차이는 32μs (2.0%)에 불과하며 실질적 성능 차이가 없습니다.
  • 4
    권장: DPDK 유지 (FTP + 멀티채널 이점). tcon 단건 통신의 RTT 개선이 필요한 경우, 네트워크 레이어가 아닌 PG Board 펌웨어 개선이 필요합니다.
