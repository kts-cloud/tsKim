Section 01

증상

  • rte_pktmbuf_alloc 호출 시 SEH Access Violation (fault address: 0xFFFFFFFFFFFFFFFF)
  • rte_eth_rx_burst, rte_eth_tx_burst도 동일한 SEH 크래시

핵심 모순: 동일한 hwio.dll + rte_*.dll 조합인데, dpdk_diag.exe에서는 정상 — ITOLED.OC에서만 크래시. DLL 자체 결함이 아닌 빌드/링킹 차이가 원인.

2.4

근본 원인 확정: __declspec(dllimport) 누락

ROOT CAUSE CONFIRMED
ROOT CAUSE

clang -shared 빌드에서 DPDK 데이터 심볼의 dllimport 누락

항목dpdk_diag.exehwio.dll
빌드 도구 meson/ninja (DPDK 빌드 시스템) clang -shared (ShimBuildStep)
링크 방식 DPDK 빌드 시스템이 관리 수동 .lib 링크
데이터 심볼 올바른 dllimport 처리 auto-import 실패

DPDK 헤더에서 문제의 데이터 심볼 선언:

// rte_mempool.h extern struct rte_mempool_ops_table rte_mempool_ops_table; ← __declspec(dllimport) 없음 // rte_ethdev.h extern struct rte_eth_fp_ops rte_eth_fp_ops[]; ← __declspec(dllimport) 없음
MECHANISM

문제의 메커니즘 (단계별)

  1. DPDK의 rte_pktmbuf_alloc은 static inline 함수 → hwio.dll에 인라인 컴파일됨
  2. 인라인 코드가 rte_mempool_ops_table (rte_mempool-26.dll의 DATA export) 접근
  3. __declspec(dllimport) 없이 컴파일 → 컴파일러가 직접 주소 참조 코드 생성
  4. lld-link(clang 링커)의 데이터 auto-import가 실패 → IAT 간접 참조 미생성
  5. 런타임에 rte_mempool_ops_table 주소로 코드 영역 주소를 읽음
  6. ops->dequeue = 가비지 (기계어 코드 바이트) → SEH Access Violation
hwio.dll (clang -shared) │ ├─ rte_pktmbuf_alloc() ← static inline, hwio.dll에 인라인됨 │ └─ rte_mempool_ops_table[ops_index] │ │ │ ├─ 정상: IAT → rte_mempool-26.dll DATA 섹션 → 유효한 ops 구조체 │ │ │ └─ 실제: 직접 참조 → 코드 영역 주소 → x86 기계어 바이트 → CRASH │ ↓ │ ops->dequeue = 0xE8C933F1... (기계어) │ ↓ │ SEH Access Violation (0xFFFFFFFFFFFFFFFF) │ ├─ rte_eth_rx_burst() ← 동일 문제: rte_eth_fp_ops 배열 └─ rte_eth_tx_burst() ← 동일 문제

영향받는 함수들:

  • rte_pktmbuf_alloc/free → rte_mempool_ops_table 접근 (alloc/free 크래시)
  • rte_eth_rx_burst/tx_burst → rte_eth_fp_ops 접근 (rx/tx 크래시)
dpdk_diag.exe가 정상인 이유: meson 빌드 시스템이 내부적으로 올바른 dllimport 처리를 하거나, EXE 링킹 시 auto-import가 정상 동작. hwio.dll은 clang -shared로 별도 빌드되어 이 처리가 누락됨.

해결 방안

3.1

FAILED

3.2

적용된 해결: hw_ring_ops.h (DPDK 헤더 이후 호출 사이트 오버라이드)

APPLIED
원리: DPDK 헤더의 인라인 함수는 이미 깨진 auto-import로 컴파일되지만, hwio.c/dpdk_netif.c의 직접 호출 사이트를 매크로로 오버라이드하여 우회.
PART 1

mempool alloc/free — 직접 ring 접근

rte_mempool_ops_table을 완전 우회. mp->pool_data (rte_ring 포인터)에서 직접 rte_ring_mc_dequeue / rte_ring_mp_enqueue 호출.

// rte_pktmbuf_alloc 대체 static inline struct rte_mbuf *_hw_pktmbuf_alloc(struct rte_mempool *mp) { void *obj = NULL; if (rte_ring_mc_dequeue((struct rte_ring *)mp->pool_data, &obj) != 0) return NULL; struct rte_mbuf *m = (struct rte_mbuf *)obj; rte_pktmbuf_reset(m); return m; } #define rte_pktmbuf_alloc(mp) _hw_pktmbuf_alloc(mp)

rte_ring_* 함수는 header-only로 ring 구조체만 접근 — 외부 글로벌 변수 참조 없음 → auto-import 문제 없음.

PART 2

rx/tx burst — GetProcAddress 런타임 해석

rte_eth_fp_ops는 배열이라 #define 우회 불가 → GetProcAddress로 런타임에 정확한 주소 취득.

static struct rte_eth_fp_ops *_hw_fp_ops = NULL; static inline void _hw_init_fp_ops(void) { HMODULE h = GetModuleHandleA("rte_ethdev-26.dll"); if (h) _hw_fp_ops = (struct rte_eth_fp_ops *)GetProcAddress(h, "rte_eth_fp_ops"); } static inline uint16_t _hw_rx_burst(uint16_t port_id, uint16_t queue_id, struct rte_mbuf **rx_pkts, uint16_t nb_pkts) { struct rte_eth_fp_ops *p = &_hw_fp_ops[port_id]; return p->rx_pkt_burst(p->rxq.data[queue_id], rx_pkts, nb_pkts); } #define rte_eth_rx_burst(p,q,r,n) _hw_rx_burst(p,q,r,n)

Section 05

교훈 및 권장사항

1

clang -shared 빌드 시 dllimport 필수

Windows에서 clang -shared로 DLL 빌드 시 DPDK와 같은 외부 라이브러리의 데이터 심볼에 __declspec(dllimport) 필요. meson 빌드 시스템 외부에서 빌드하면 자동 처리 안 됨.

2

DPDK 버전 업그레이드 시 호환성 확인

향후 DPDK 버전 업그레이드 시 hw_ring_ops.h의 rte_eth_fp_ops 구조체 레이아웃 호환성 확인 필요. 특히 rxq.data[], rx_pkt_burst 필드 위치.

3

hwio.dll은 drivers.dat에만 포함

hwio.dll은 drivers.dat 안에 포함되어 런타임에 temp dir로 추출 → SetDllDirectory 경유 로드. 앱 실행 폴더에 hwio.dll을 별도로 두면 Windows DLL 검색 순서상 앱 폴더가 우선되어 구버전이 로드될 위험이 있음. 배포 시 앱 폴더에는 drivers.dat + HwNet.dll만 복사.

4

LoadLibraryA 전체 경로 vs 파일명만

LoadLibraryA("C:\temp\hw_xxx\hwio.dll") (전체 경로)로 로드하면 .NET P/Invoke의 [DllImport("hwio.dll")] 해석 시 모듈명 불일치로 DLL을 찾지 못함. SetDllDirectory 설정 후 LoadLibraryA("hwio.dll") (파일명만)으로 로드해야 P/Invoke와 일치.
